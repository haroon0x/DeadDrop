A receipt should be evidence, not agent prose

Agent summaries are useful. Agent claims about files, tests, and success are not enough to operate a reliable queue.

When a coding agent finishes, it naturally explains what it believes it did. That explanation is valuable: it can describe intent, tradeoffs, blockers, and decisions that are not obvious from a patch. It is also produced by the same fallible process that made the change.

If a job system copies that explanation directly into “changed files” and “tests passed,” it turns a narrative into an audit record. The interface looks structured, but the underlying trust model has not improved.

Split subjective and observable fields

DeadDrop asks the agent for a structured JSON receipt with a summary, notes, blockers, changed files, verification, and status. The worker validates the shape, then deliberately replaces the fields it can observe.

FieldAuthority
SummaryAgent
Notes and blockersAgent
Changed filesGit diff
VerificationWorker commands
StatusObserved exit state

This keeps the best part of agent output—the explanation—without asking it to certify itself.

Git is the changed-file authority

The worker already has a job baseline and isolated worktree. It can ask Git which paths differ from that baseline. That answer includes files the agent forgot to mention and excludes files it claimed to change but did not.

The same principle applies to the patch. The dashboard does not reconstruct edits from a model response. It stores the binary Git diff produced after execution.

Verification must be configured outside the prompt

A prompt can ask an agent to run tests, but the worker cannot infer which command is authoritative for every repository. The manifest therefore accepts trusted verification commands:

{
  "alias": "default",
  "path": "/projects/api",
  "verify": ["pytest -q", "ruff check ."]
}

The worker runs those commands after the agent in the same isolated workspace. It records the exact command, pass or fail status, and observed exit code. Any failure fails the job. If no commands are configured, the receipt says no verification was reported instead of preserving an agent's unsupported claim.

Configuration is a trust boundary. Verification commands are local operator input and can execute arbitrary code. They do not come from the remote task.

Status is an outcome, not a tone

An agent may sound confident after a non-zero exit, or produce a valid receipt before a later verification failure. The worker sets receipt status from the final observed state: completed for zero, cancelled for the cancellation path, and failed otherwise.

A zero-exit agent that omits a valid structured receipt also fails. That rule prevents silent degradation into an unstructured output blob that the interface cannot interpret honestly.

Receipts become useful when fields have provenance

The goal is not to distrust every sentence an agent writes. It is to avoid asking one source to be both actor and auditor. Summaries explain. Git enumerates. Commands verify. Process state decides. A receipt becomes credible when each field comes from the narrowest authority capable of producing it.
