Leases make local agents recoverable

A local worker can sleep, lose Wi-Fi, crash, or return late. A durable queue needs ownership that expires safely.

The simplest job queue has two operations: fetch the next queued row and mark it running. That works until the process disappears between those operations and completion. The database now contains a running job that no worker owns, and “running” has become a permanent state.

Local coding workers make this especially common. Laptops suspend. Networks change. Agent CLIs hang. Users kill terminals. A reliable design must treat worker disappearance as normal, not exceptional.

An attempt owns the job temporarily

When a DeadDrop worker claims a job, the server generates a unique attempt ID, increments the attempt number, and assigns a lease expiration. The worker includes that attempt ID with every log, heartbeat, and terminal result.

job 42
status: running
attempt_number: 3
attempt_id: 7f9b…
lease_expires_at: 12:01:00Z

The attempt ID is more important than the running flag. It answers “which execution currently owns the right to write?”

Heartbeats renew, polls recover

The worker sends a heartbeat every five seconds. A successful heartbeat moves the sixty-second lease forward and updates both the job and attempt history.

Recovery does not require a separate scheduler. Before the server claims another queued job, it scans for running jobs with expired leases. Each expired attempt becomes lost. Its job returns to queued unless a cancellation was already requested.

The next claim receives a new attempt ID. If the old worker reconnects and submits its result, the server compares IDs and returns a conflict. The late process cannot overwrite newer work.

Why not retry the claim request?

HTTP retries are usually good, but claim is a dangerous boundary. The server might commit the claim while the response is lost. Retrying immediately could claim another job while the first remains running.

DeadDrop retries idempotent registration, logs, heartbeats, and terminal delivery. It does not retry claim. The lease is the recovery mechanism for an ambiguous claim response.

Cancellation belongs inside the ownership protocol

Cancelling a queued job is a database transition. Cancelling a running job requires the current attempt to cooperate. The owner request sets a cancellation timestamp. The next heartbeat returns that signal to the active worker.

The worker cancels its command context and kills the spawned process group, then captures any partial patch and acknowledges a cancelled result. If completion races with the owner request, the server makes cancellation win while still accepting the attempt's final evidence.

Cancellation is not a second control channel. It rides the same outbound heartbeat path, preserving the no-inbound-connection design.

Terminal delivery needs its own durability

A job can finish locally while the server is unreachable. Retrying helps with short failures, but a laptop should not lose the result when the retry budget ends.

The worker writes the terminal payload atomically to a user-only pending-results directory. Before claiming more work, it replays that queue. Completion and failure endpoints are idempotent for the same attempt, so a lost success response does not create a conflict on replay.

If the server permanently rejects a stale or invalid payload, the worker preserves it as a rejected record instead of blocking every future job. Authentication and transient errors continue to block because they can be corrected without losing ownership history.

The queue becomes understandable under failure

Leases do not prevent machines from failing. They turn failure into explicit state: running attempt, renewed lease, lost attempt, new attempt, rejected stale write. That vocabulary makes both code and operator behavior predictable.

For local agents, recoverability is not an enterprise add-on. It is the minimum needed to turn a promising demo into a queue someone can leave unattended.
